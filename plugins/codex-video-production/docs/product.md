# 产品设计：TVC 广告视频完整生产

## 产品目标

把用户不成熟的制作要求和可选产品图、链接、品牌资料，推进为可审阅、可恢复、可追溯的完整 TVC 成片。Codex 是唯一编排者；用户在本地 Preview 看到真实制作文件并在关键节点做选择、批注和批准。

产品不是自动工作流平台、通用视频编辑器或第二套 Agent Runtime。浏览器不生成媒体、不决定下一阶段，也不替用户推断审批。

## 目标用户旅程

1. 用户输入模糊需求，可附产品图、品牌资料、链接或文案。系统把原文写入 `request.md`，把真实素材导入 `assets/` 并登记。
2. 需求澄清由 Host Agent 根据本次用户目的和已有素材动态生成，尽量在 Preview 主内容区以一次单选、多选和必要文本组成的完整问卷完成：问题纵向排列、选项使用可直接点击的胶囊样式，只询问会改变研究、创意或交付的缺口，结果形成确认 Brief。
3. Host Agent 自主调用研究工具，完成有来源的产品、品类、受众和竞品研究。
4. Host Agent 先完成 3–4 个显著不同的完整文字策划方向，并为每个方向各生成一张覆盖全片的概念故事板。每个镜头格严格使用 Brief 已确认的成片画幅；整张故事板的镜头数量由叙事节奏决定，Host 再根据镜头格画幅、数量和标注空间推理行列与整图比例。Preview 顶部用文字方向卡呈现差异；点击卡片即选中并在下方只展开该方向的一张全片故事板和完整文字提案，页面底部一次确认继续。不存在额外“选择这个方向”按钮，也不存在先选文字、再等图片、再确认一次的中间状态。文字提案仍是创意语义权威，概念故事板负责让普通人员直观理解全片事件、节奏与产品角色。
5. 方向确认后，Host Agent 在“分镜制作”阶段内部完成视频结构、跨镜头人物/产品/场景参考图和分镜提示词。Generation Unit 是一次视频模型生成的片段，不等于镜头、卖点或故事板面板；结构优先使用单次 15 秒能力并把 Unit 数降到最少，30 秒 TVC 默认通常为两段约 15 秒，60 秒通常为四段。只有主体、场景、时间或连续性等真实生成边界才增加或缩短 Unit，并在结构中记录原因。每 Unit 连续无重叠、总和精确等于成片时长，并声明 tail-frame、match-cut 或 reference-only 衔接。这些内部文件可审计，但不要求普通用户逐项批准。
6. 每 Unit 一次调用生成一张覆盖完整时间进程、不可拆分的 Storyboard Sheet。每个镜头格的可见画面区域必须严格等于成片画幅，例如 9:16 成片中的每格都是竖幅 9:16，而不是在横幅镜头旁标注“9:16”。面板数量由该 Unit 的叙事密度决定；数量确定后，Host 根据镜头格比例、行列利用率和标注可读性推导整张 Sheet 的比例，整图比例不要求等于成片比例。默认使用黑白手绘铅笔故事板而非写实成片效果：画面只保留必要的彩色运动、运镜、构图、灯光、声音/卖点和中文台词标注。用户提供产品图时，每个 Unit 都把至少一张产品图作为真实生成参考，以保护跨段产品一致性。Preview 先用文字制作卡一次呈现所有 Unit 的时间、叙事作用、画面事件与衔接关系；点击某张卡后，只展示该 Unit 的整张分镜、图片生成提示词和参考素材文字清单，原素材按需打开而不直接铺图。未标记的段落默认通过，只选择需要修改的整张分镜并填写意见。
7. “视频片段制作”阶段内部为每 Unit 建立 Prompt Plan；Storyboard 是第一张图片引用，编号与真实引用数组一一对应。Plan 必须把一张多面板 Storyboard 翻译成带时间点的多镜头连续过程，不能退化成一次缓慢推镜、重复动作或长时间产品定格。常规高能产品 TVC 的多数镜头约 0.8–3 秒，只有已选创意明确要求长镜头时才延长。付费视频调用前必须存在该 Unit 当前 Storyboard 的有效批准证据，Plan 本身不单独要求用户批准。
8. 独立 Unit 先全部提交、统一轮询，全部生成完成后一次批量审阅。每个 Unit 在业务界面只显示一个当前视频；不满意的段落填写意见并重做。仅真实 tail-frame 依赖链局部串行。
9. 结构审计通过后，以 ffmpeg 按 Unit 顺序确定性合成，探测真实交付属性并在 Preview 完成成片批准。

## 用户可见物与决策点

| 环节 | 用户可见物 | 决策 |
| --- | --- | --- |
| 澄清 | 主内容区完整问卷、原始需求、素材清单、Brief 摘要 | 提交表单 |
| 调研 | 来源、事实/客户声明/推断、受众与竞品发现 | 无例行审批 |
| 提案 | 3–4 张文字方向卡；当前卡对应的一张全片概念故事板和永久展开的完整文字提案 | 点击卡片即选中方向，底部一次确认继续 |
| 分镜制作 | 已确认的全片方向、全片 Unit 文字制作卡；当前卡对应的规范故事板、图片提示词和参考素材 | 点击卡片切换单段详情；默认全部通过，只标记需修改的分镜并填写意见；结构与提示词不单独审批 |
| Unit 视频 | 每段当前视频、Storyboard 对照、衔接信息、生成状态 | 批量批准/重做 |
| 制作包审计 | 路径化问题和修复建议 | 无例行审批 |
| 成片 | 成片播放器、时长/比例/编码事实、Unit 构成 | 最终批准 |

Preview 面向传媒公司的普通业务与制作人员，默认使用业务名称和自然语言，不暴露文件路径、JSON、Git、MCP、Stage 或 Task 等内部术语。需求澄清问卷占用主内容宽度；创意提案及后续环节使用各自独立的业务页面，内容区不重复显示阶段步骤条。概念与正式故事板始终作为整张图片缩放、查看和整图选择，UI 不裁切或重组其中的面板。制作流程默认收起；展开后可以只读回看已完成环节，未来环节只显示名称和顺序但不可打开。修改一个 Unit 只使该目标的批量审批失效；恢复完全相同的原字节会恢复原审批。

## 验收标准

- 中断或 MCP 重启后，以同一 `outputDir + productionId` 恢复，不丢用户提交、任务 ID、提交 ID或结果来源。
- 任何 required Review 缺少当前 Git 对象证据时，Stage 不能完成。
- 当前分镜未批准、批准已失效或未作为任务依赖时，不能创建付费视频任务；内部 Plan 可随执行需要更新，但必须保持已提交和可追溯。
- 8-Unit 全流程不传输完整制作快照；状态、账本和任务均用窄读写，传输 token 相比整本读写至少下降 10 倍。
- Storyboard 与每段当前 Clip 为真实本地文件，Preview 可显示/播放；任务留存文件不重复展示；最终成片时长、比例与 Brief 一致。
- 提案选择时所有方向都已有可见的全片故事板和完整文字；正式 Storyboard 使用黑白手绘规划语言并包含约定的彩色标注系统，不以写实成片板代替；存在产品参考图时，Plan、Task 依赖与 Preview 素材清单都能追溯其实际使用。
- 默认音频范围是视频模型原生音频和用户提供音频；音乐生成、TTS 配音和独立音效生成不在当前范围。
- 所有媒体调用只复用 Lovart Skill；项目不复制 Lovart、videoagent 或 OpenMontage 运行时代码。
