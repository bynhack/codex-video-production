# TVC — Storyboard Production

## Stage goal

Internally turn the selected creative direction into executable Generation Units, establish shared continuity references, generate one hand-drawn Storyboard Sheet per Unit, and let the user mark only the Units that need changes.

## Required inputs

Read the confirmed Brief, selected complete text Proposal, its whole-film Concept Storyboard and every usable source asset. The text Proposal remains semantic authority and the selected Concept Storyboard carries whole-film visual intent.

## Domain instructions

First write the minimum safe ordered Unit structure. A Unit is one Provider generation clip, not one shot, selling point, visual beat, scene, action or Storyboard panel. Minimize Unit count and use as much of the Provider's 15000ms capacity as the story can safely sustain: a 30000ms film normally uses two approximately 15000ms Units, and a 60000ms film normally uses four. Add shorter or additional Units only when a hard generation boundary makes one continuous clip unreliable, such as an incompatible subject/scene/time change, a required real tail-frame dependency, a Provider duration limit, or an explicit user request. Record that concrete reason in `structure.json`; narrative variety by itself is not a reason to split. Units are contiguous, non-overlapping, greater than zero, at most 15000ms and sum exactly to total duration. Define `tail-frame`, `match-cut` or `reference-only` continuity. Identify shared character, product and scene identities; reuse real user assets when sufficient and otherwise use only `lovart-unofficial` for the minimum reference images.

For each Unit create one self-contained plan for one indivisible Storyboard Sheet covering the full Unit chronologically. If the user supplied any asset classified as `role: product-reference`, every Unit Plan must include at least one such committed local image in `referencePaths`; pass those exact files to Lovart when generating the Storyboard. The reference still protects identity continuity when the product is temporarily off-screen. Never substitute the whole-film Concept Storyboard for the available product identity reference. When no product reference exists, record that fact in the Plan instead of silently pretending one was supplied.

Panel count follows this Unit's narrative density; do not hard-code a shot count. Panels are the multiple shots and temporal beats inside one generated Unit, not reasons to create more Units. Set `frameAspectRatio` exactly to the confirmed delivery ratio in `brief.md` and `structure.json`. Every panel's visible image aperture—not merely a label—must use that ratio, so a 9:16 delivery produces portrait 9:16 panels throughout. Put time ranges and production annotations outside those apertures. Once panel count is known, infer a readable `columns × rows` grid with minimal unused cells, derive the ideal whole-sheet geometry from the grid and panel ratio, and choose the nearest Lovart-supported `sheetAspectRatio`; the sheet itself need not match the delivery ratio. Record this layout reasoning and state the panel ratio, grid and sheet ratio explicitly in the generation prompt. Never widen portrait panels into horizontal strips, stretch them, or mix panel ratios to fill the page.

The drawing itself is black-and-white rough pencil line art with minimal detail, rapid gesture energy, simple anatomical construction and strong silhouette readability—not photorealistic frames, glossy renders or a dark finished-ad layout. Every panel shows an event in progress with state change, physical or energy movement, and camera participation; forbid explanatory feature diagrams, parameter lists, static product displays, UI and HUD. Build emotional escalation and small narrative closures, normally grouping roughly three panels into a beat when density permits. Give every panel a real subrange inside the Unit and design an intentional edit rhythm rather than stretching one action across the clip. For a conventional energetic product TVC, prefer several concise shots and visible changes, with most shots roughly 0.8–3 seconds and no decorative hold longer than 4 seconds; deviate only when the selected Proposal explicitly calls for a sustained shot. Include time ranges and Chinese annotations: red arrows for body/object motion, blue arrows for camera motion, green text for framing/composition, orange marks for light direction, purple text for sound/selling-point emotion, and black text for dialogue, using `-` when there is no dialogue. Derive visual language, camera behavior, environment, energy and pacing from the selected Proposal, never from a fixed house style. The Proposal is the authority: a calm, minimal, warm or restrained direction must produce calm, restrained storyboards, and only a direction that genuinely calls for high energy should reach for aggressive techniques such as whip pan, telephoto compression, handheld energy, smoke, hard light beams or wet-floor reflections. Let the Proposal's own narrative arc decide where any climax, energy shift, or final product/brand lock-up belongs; do not impose a fixed quota of climaxes or camera impacts. Across each Unit still require a meaningful change from start state to end state so the clip has real motion to generate.

## Output contract

Write `structure.json` with exact timing and ordered Units. Write `continuity/index.json` with `references[]` containing stable ID, kind, real path, source paths and used Unit IDs. For each Unit write `units/<unitId>/storyboard-plan.json`, generate candidates under `storyboard-candidates/*.png`, and copy the Host-selected current candidate bytes to `storyboard.png` before user review. Plans include Unit timing, `frameAspectRatio`, dynamic `panelCount`, `grid` with positive `columns` and `rows`, `sheetAspectRatio`, `layoutRationale`, the complete style/layout/annotation prompt, ordered real `referencePaths`, `continuityInputs`, and empty `blockingIssues`. Do not persist panel crops.

## Review and completion

Commit internal structure, continuity references, plans, candidates and current canonical Storyboards. Every Storyboard image Task uses the Unit Plan as `requestRef` and stamps every `referencePaths` entry as a dependency before the Lovart call. Request one `batch-review` targeting each `units/<unitId>/storyboard.png` in Structure order. Preview treats every target as approved by default; the user marks only images that need modification and supplies target-specific feedback. Append and commit the Decision. Regenerate and re-review only revised targets. Complete when timing and references are valid, every canonical Storyboard is byte-identical to a successful candidate or imported source, and every current canonical path has an `approve` action. Pause video execution behind `tvc.unit-video-production@1`.

## Boundaries

Do not ask users to approve Structure, reference planning, prompts or candidate directories. Do not generate video/audio, let Preview call Providers, use any media skill other than Lovart, or replace the hand-drawn planning-board style with photorealistic presentation frames.
